# XXL-Job

## 重点概念

### 调度中心（Admin）

- 作用

	- 调度中心主要有两个工作：1.定时读取注册的执行器，根据调度策略和路由策略将任务分配给这些执行器去执行。2.管理系统日志

- 子组件

	- 触发器

		- 作用

			- 触发器是具体去调用远程执行器的线程。调度中心维护了一个触发器线程的线程池。根据调度策略，当一个任务需要被执行时，调度中心将这个任务交给一个触发器去触发远程调用。触发器再根据路由规则去调用远程执行器。实现了调度和调用的解耦。

### 执行器（Excutor）

- 作用

	- 执行器主要有4个工作：1.获取远程调度中心地址，将自己注册进去。2.作为rpc的服务端，接受调度中心的远程调用，基于线程池执行任务。3.作为rpc的客户端，任务结束后，调用调度中心的远程回调方法。4.接受调用中心的日志请求，读取日志文件并返回结果。

### HA/集群

- 部署xxl-job的时候，可以部署多台调度中心，它们会使用同一个MySQL实例，并且使用数据库锁的方式实现分布式锁。同时，执行器也可以集群化部署，调度中心是通过App name来区分一组执行器实例的。也就是说，配置同一个App name的所有执行器实例视为同一个执行器，调度中心将一个任务交给一组执行器实例去执行。

### 通信

- 调度中心与执行器之间的通信是全部异步化的，从任务的触发、执行、到结果回调。消息可以入队的话，都是通过队列的方式来完成的。通信请求都是使用各自的代理对象中的方法，进行远程调用。
- 注册

	- 每个执行器会以心跳注册的方式向调度中心维护自己的注册状态，如果连续三次心跳中断的话，调度中心会摘除当前的执行器。

- 触发

	- 调度中心执行任务时，会去通过触发器触发对应的执行器。调度中心启动后会维护一组触发器线程池，从线程池中取出一个触发器，让它处理相关的远程调用。

- 回调

	- 每个执行器都维护了一个回调队列和回调线程。任务执行完成后会将返回结果放入回调队列中，回调线程循环从队列中取出结果，调用调度中心的代理对象中的回调方法，进行远程回调。

### 分片任务

- 通过分片广播的方式实现任务分片。分片任务指派给一个执行器后，会对这个执行器下所有实例进行排序，每一个实例就拥有了自己的序号，并且也知道实例的总数，这两个参数作为分片参数传递给每个实例。执行器实例再根据自己得到的分片参数执行相应的业务逻辑。

### 阻塞策略

- 进行一些耗时的任务时，可能一个任务触发的频率超过它的执行器执行的速度，这时我们就需要处理这些堆积的触发请求。阻塞策略对于同一个任务路由到同一个执行器的情况才有效。
- 单机串行

	- 触发请求会入队，执行器按先后顺序从队列中取出这些触发请求。

- 丢弃后续调度

	- 如果当前任务已经在执行了，就把后续进来的任务标记为失败。

- 覆盖之前调度

	- 新来的任务会把正在执行的任务和队列里等待的任务都标记为失败，然后执行新来的任务。

### 失败策略

- 任务的失败会进行告警处理。默认为邮箱告警，可自行扩展。

### 路由策略

- 详见源码分析。

### 触发规则

- 触发规则主要有3种

	- Cron表达式

		- 定时执行任务。

	- 任务依赖

		- 父任务执行完成后，可以触发子任务。需要任务依赖关系可以配置。

	- 事件触发

		- 类似MQ的场景，可以自行在业务逻辑中触发任务。

### 任务模式

- Bean模式

	- 本质是Spring容器中的一个Bean，添加相关注解，会自动被执行器扫描找到。

- Glue模式

	- 可以在服务正在运行的时候动态地添加业务逻辑，类似热更新。基于web IDE进行开发，无需重新打包上线，支持版本回溯。

## 源码分析

### 项目依赖关系

- admin项目和excutor项目共同依赖core项目。

### 公共依赖的项目中重要的组件

- 远程调用代理对象

	- 作为代理对象，用于远程调用
	- AdminBizClient

		- 实现了AdminBiz接口，执行器可以用它进行注册，撤销注册和回调等远程调用。

	- ExecutorBizClient

		- 实现了ExecutorBiz接口，调度中心可以用它对执行器进行心跳检测等远程调用。

- 日志管理

	- XxlJobLogger

		- 写日志到文件。

### 调用中心的重要组件

- 线程与线程池

	- JobRegistryMonitorHelper

		- 用于监控执行器注册的线程。定期查询数据库，读取注册的执行器信息。

	- JobFailMonitorHelper

		- 用于监控失败日志，进行任务失败处理的线程。

	- JobLosedMonitorHelper

		- 用于任务结果丢失处理的线程

	- JobTriggerPoolHelper

		- 触发器线程池，用于维护一组线程。有任务需要触发时，从中取出一个线程来处理。
		- XxlJobTrigger

			- 线程池中的触发器线程会调用这个类的trigger方法，触发远程的执行器。

	- JobLogReportHelper

		- 运行报告监控器线程

	- JobScheduleHelper

		- 定时任务调度器线程。定期从数据库中读取定时任务，交给触发器线程池处理。

- 路由策略

	- 具体路由策略

		- ExecutorRouteRound

			- 轮询策略，维护了一个map，记录每个job进行路由的计数，每次路由时取出计数并++，与注册执行器地址列表的size取余。这样可以保证每个任务能轮询一遍注册执行器列表。

		- ExecutorRouteRandom

			- 随机策略，随机从注册执行器的地址列表取出一个地址。

		- ExecutorRouteFirst

			- 第一个策略，返回注册执行器的地址列表的第一个地址。

		- ExecutorRouteLast

			- 最后一个策略，返回注册执行器的地址列表的最后一个地址。

		- ExecutorRouteConsistentHash

			- 一致性hash策略，分组下机器地址相同，不同JOB均匀散列在不同机器上，保证分组下机器分配JOB平均；且每个JOB固定调度其中一台机器。

		- ExecutorRouteLRU

			- 最近最久未使用策略，为每个任务维护了一个LinkedHashMap记录访问的执行器的地址顺序，按顺序最久未使用的优先。

		- ExecutorRouteLFU

			- 最近不经常使用策略，为每个任务维护了一个HashMap，对每个任务记录每个地址使用的次数。取地址时将map转换成ArrayList进行排序，取使用次数最少的地址。

		- ExecutorRouteBusyover

			- 故障转移策略，对执行器进行心跳检测，通过的则返回这个执行器地址，不通过的则检测下一个执行器。

		- ExecutorRouteFailover

			- 忙碌转移策略，检测执行器是否正在执行任务，是则跳过，检测下一个，否则返回这个执行器地址。

	- 路由策略总结

		- 设计模式使用了策略模式，每种路由策略都继承了ExecutorRouter这个抽象父类，并重写父类的route抽象方法。调用方可以调用route的实现方法来获得注册的执行器的路由地址。

### 执行器的重要组件

- 线程与线程池

	- ExecutorRegistryThread

		- 开启一个线程，每隔30秒向调度中心请求注册，维持自己的注册

	- EmbedServer

		- 启动一个线程，监听一个端口，处理来自调度中心的rpc请求。

	- TriggerCallbackThread

		- 启动一个线程，进行任务执行完成后的回调工作。它维护了一个回调队列，循环从中取出任务执行结果。使用调度中心的代理对象中的callback方法进行远程调用。

- 任务执行相关

	- IJobHandler

		- 抽象类，可根据业务需求自行实现的execute方法。可用于任务执行模式中的Glue模式。

	- ExecutorBizImpl

		- 实现了ExecutorBiz接口，作为执行器的另一个代理对象，可以被本地的EmbedServer监听线程调用，进行心跳检测、任务执行等工作。其中任务执行主要有2个步骤。1.向XxlJobExecutor申请一个JobThread线程来执行任务。2.将放入JobThread的执行队列中

	- XxlJobExecutor

		- 控制执行器生命周期的实现类。维护了一个map作为注册表，每个任务id对应了一个处理它的JobThread。多个任务可以对应一个JobThread。同时维护了另一个map，用来注册handler的名字和对应的handler。
		- JobThread

			- 维护了一个执行队列，聚合了一个IJobHandler，循环从队列中取出任务来执行。执行时，再启动一个线程阻塞地执行任务，调用handler的excute方法。执行完毕，将结果放入TriggerCallbackThread的回调队列中。
